代码改了**必须重新 `npm run build`** 才生效。

### 步骤 1-2：本地改 + 提交（同上）

```bash
# SSH 登录
ssh ubuntu@124.222.151.69

# 拉取代码
cd ~/gx-project/frontend
git pull

# 如果 package.json 变了，先装依赖
npm install --no-audit --no-fund
```

> Nginx 的 root 已指向 `~/gx-project/frontend/build`，build 完成即刻生效，无需复制文件。

### 步骤 4：验证前端可访问

```bash
curl -I http://127.0.0.1/
# 应返回 200 OK

# 公网验证
curl -I http://124.222.151.69/
# 应返回 200 OK
```

### ⚠️ 前端 build 失败怎么办

```bash
# 1. 看 npm 报错，常见问题：
#    - node_modules 缺失 → npm install
#    - 内存不足 → 加 --max-old-space-size=4096
#    - 类型错误 → 回去改代码
#
# 2. 实在不行，临时回滚到上一个 build：
cd ~/gx-project/frontend
git log --oneline | head -5   # 找上一个稳定版本
git checkout <commit-hash>
```

## 四、数据库变更流程

> 📌 **库说明**
> - `gx_project`：**生产库**，本次更新涉及
> - `gx_project_dev`：**开发库**，仅本地/测试用，不参与生产部署
> - 数据库运行在 Docker 容器 `gx_mysql` 中，命令统一用 `docker exec`

### 仅改数据（INSERT/UPDATE/DELETE）

```bash
# 直接连上去改
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project

# 或者写 SQL 文件
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project < your-changes.sql
```

### 改表结构（DDL）

**必须**经过评审 + 备份，流程：

```bash
# 0. 确保备份目录存在（首次执行时目录不存在，需先创建）
mkdir -p /home/ubuntu/backup

# 1. 备份当前数据库（强制）
#    仅备份生产库 gx_project（gx_project_dev 是开发库，不需要备份）
sudo docker exec gx_mysql mysqldump -uroot -pGX2026\!root \
  --single-transaction --routines --triggers \
  gx_project > /home/ubuntu/backup/gx_project_$(date +%Y%m%d_%H%M%S).sql

# 2. 先在开发库 gx_project_dev 测试 DDL
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project_dev \
  < schema-changes.sql

# 3. 测试通过后，在生产库执行
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project \
  < schema-changes.sql

# 4. 如果后端有依赖新字段的代码改动 → 先改表，再重启后端
#    如果只是默认值变化 → 改完表就行
```

## 六、部署后验证清单（强制，不可跳过）

> ⚠️ **教训**：curl 测接口返回 200 ≠ 用户能正常使用。
> curl 绕过了浏览器的 CORS、JS 执行、连锁请求，**用 curl 测通不代表部署成功**。
> 每次部署后必须完成以下全部验证，才能对用户宣布"部署完成"。

> 📖 **测试方法论与开发环境验证**详见 [测试规范.md 第五章](../07-测试规范/测试规范.md) —— 本章是**部署场景专用清单**,测试规范是**通用方法论**(开发+部署)。

### 6.1 验证方法分级（必须区分并如实报告）

| 级别 | 方法 | 能发现 | 不能发现 |
|------|------|--------|---------|
| L1 服务存活 | `systemctl is-active` / `docker ps` | 进程是否在跑 | 功能是否正常 |
| L2 接口连通 | `curl` 打单个接口看状态码 | 接口能否响应 | **CORS、前端调用、登录后连锁请求** |
| L3 端到端 ✅ | **真实/无头浏览器走完整用户路径** | CORS、白屏、连锁500、前端报错 | — |

**只有 L3 通过才算"部署完成"。L1/L2 通过只能说"服务已启动/接口可达"，不得表述为"可以正常使用"。**

### 6.2 端到端验证清单（逐项打勾）

```
[ ] 1. 浏览器打开 http://124.222.151.69/ ，首页正常渲染（非白屏）
[ ] 2. F12 控制台无红色报错（重点看 CORS / Network Error）
[ ] 3. 用 admin / Admin@2026 实际登录，能进入系统
[ ] 4. 登录后首页 Dashboard 数据正常加载（overview/todos 等连锁接口无500）
[ ] 5. 逐个点开核心页面（项目计划/学校/风险等），无报错
[ ] 6. 换一个非 admin 账号登录，验证权限正常
```

**推荐用 Playwright 自动化执行**（项目已有 `playwright-ui-auto` skill）——它是真实浏览器，能自动发现 CORS、白屏、连锁 500，比人工点更可靠。

### 6.3 CORS 专项检查（前后端分离必查）

```bash
# 带 Origin 头模拟浏览器跨域，检查响应是否有 Access-Control-Allow-Origin
curl -s -I -X OPTIONS http://124.222.151.69/api/v1/auth/login \
  -H "Origin: http://124.222.151.69" \
  -H "Access-Control-Request-Method: POST" | grep -i "access-control"
# 无输出 = CORS 未放行，浏览器会拦截（即使 curl 直接打是 200）
```

### 6.4 数据库 schema 对账（部署前置步骤，非事后补救）

代码模型、ddl.sql、生产库三方必须一致，否则会出现"缺列导致 500"。**上线前**执行：

```bash
# 用最新 ddl.sql 建临时库，与生产库逐表对比列差异
docker exec gx_mysql mysql -uroot -pGX2026\!root -e \
  "DROP DATABASE IF EXISTS gx_schema_check; CREATE DATABASE gx_schema_check;" 
awk '/^USE /{next} /^CREATE DATABASE/{s=1} s&&/;/{s=0;next} s{next} {print}' ddl.sql \
  | docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_schema_check
# 逐表 comm 对比 gx_schema_check 与 gx_project 的列，缺列/多列都要处理
```

### 6.5 状态报告纪律

- ✅ 允许："L2 接口验证通过；浏览器端到端(L3)验证通过，登录及首页正常"
- ❌ 禁止：只做了 curl 就说"部署完成，可以使用了"
- 有未验证项必须**明说**："X 已验证，Y 未验证"，不含糊带过

## 八、变更窗口建议

- **后端更新**：低峰期（晚上 22:00 后）重启会有 1-2 秒 502
- **前端更新**：任何时间，build 期间前端可访问旧的 build

## 九、问题排查

| 现象 | 排查方向 |
|---|---|
| 502 Bad Gateway | `systemctl status gx-backend` → 看进程是否在 |
| 前端页面空白 | `curl -I http://127.0.0.1/` → 看 Nginx 状态 |
| 数据库连不上 | `docker ps` → 容器在不在；`docker logs gx_mysql` |
| 本地开发连不上远程库 | `telnet 124.222.151.69 3306` → UFW 白名单检查（见部署手册 10.4） |
| 前端显示旧版本 | 浏览器清缓存（Ctrl+Shift+R）；或重新 `npm run build` |
| 后端 API 500 | `journalctl -u gx-backend -n 50` → 看 Python 报错 |
| CORS 跨域错误 | 检查 `backend/app/main.py` CORS 配置 + **必须重启后端**（见部署手册 10.5） |
