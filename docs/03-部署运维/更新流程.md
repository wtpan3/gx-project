```bash
docker ps                                    # 确认 gx_mysql 容器运行中
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root -e "SELECT 1"
```

## 三、前端更新流程

### ⚠️ 重要：生产是 build 模式，非热更新

代码改了**必须重新 `npm run build`** 才生效。

### 步骤 1-2：本地改 + 提交（同上）

### 步骤 3：服务器拉取 + 重新 build

```bash
# SSH 登录
ssh ubuntu@124.222.151.69

# 拉取代码
cd ~/gx-project/frontend
git pull

# 如果 package.json 变了，先装依赖
npm install --no-audit --no-fund

# 重新 build（覆盖 /home/ubuntu/gx-project/frontend/build/）
npm run build

# 验证 build 产物
ls -la build/
# 应看到 index.html + static/ 目录
```

> Nginx 的 root 已指向 `~/gx-project/frontend/build`，build 完成即刻生效，无需复制文件。

### 步骤 4：验证前端可访问

```bash
curl -I http://127.0.0.1/
# 应返回 200 OK

# 公网验证
curl -I http://124.222.151.69/
# 应返回 200 OK
```

### ⚠️ 前端 build 失败怎么办

```bash
# 1. 看 npm 报错，常见问题：
#    - node_modules 缺失 → npm install
#    - 内存不足 → 加 --max-old-space-size=4096
#    - 类型错误 → 回去改代码
#
# 2. 实在不行，临时回滚到上一个 build：
cd ~/gx-project/frontend
git log --oneline | head -5   # 找上一个稳定版本
git checkout <commit-hash>
npm run build
```

## 四、数据库变更流程

> 📌 **库说明**
> - `gx_project`：**生产库**，本次更新涉及
> - `gx_project_dev`：**开发库**，仅本地/测试用，不参与生产部署
> - 数据库运行在 Docker 容器 `gx_mysql` 中，命令统一用 `docker exec`

### 仅改数据（INSERT/UPDATE/DELETE）

```bash
# 直接连上去改
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project

# 或者写 SQL 文件
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project < your-changes.sql
```

### 改表结构（DDL）

**必须**经过评审 + 备份，流程：

```bash
# 0. 确保备份目录存在（首次执行时目录不存在，需先创建）
mkdir -p /home/ubuntu/backup

# 1. 备份当前数据库（强制）
#    仅备份生产库 gx_project（gx_project_dev 是开发库，不需要备份）
sudo docker exec gx_mysql mysqldump -uroot -pGX2026\!root \
  --single-transaction --routines --triggers \
  gx_project > /home/ubuntu/backup/gx_project_$(date +%Y%m%d_%H%M%S).sql

# 2. 先在开发库 gx_project_dev 测试 DDL
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project_dev \
  < schema-changes.sql

# 3. 测试通过后，在生产库执行
sudo docker exec -i gx_mysql mysql -uroot -pGX2026\!root gx_project \
  < schema-changes.sql

# 4. 如果后端有依赖新字段的代码改动 → 先改表，再重启后端
#    如果只是默认值变化 → 改完表就行
```

## 七、变更窗口建议

- **后端更新**：低峰期（晚上 22:00 后）重启会有 1-2 秒 502
- **前端更新**：任何时间，build 期间前端可访问旧的 build
- **数据库 DDL**：必须低峰期，并先备份
- **重大变更**：提前通知项目成员

## 八、问题排查

| 现象 | 排查方向 |
|---|---|
| 502 Bad Gateway | `systemctl status gx-backend` → 看进程是否在 |
| 前端页面空白 | `curl -I http://127.0.0.1/` → 看 Nginx 状态 |
| 数据库连不上 | `docker ps` → 容器在不在；`docker logs gx_mysql` |
| 本地开发连不上远程库 | `telnet 124.222.151.69 3306` → UFW 白名单检查（见部署手册 10.4） |
| 前端显示旧版本 | 浏览器清缓存（Ctrl+Shift+R）；或重新 `npm run build` |
| 后端 API 500 | `journalctl -u gx-backend -n 50` → 看 Python 报错 |
| CORS 跨域错误 | 检查 `backend/app/main.py` CORS 配置 + **必须重启后端**（见部署手册 10.5） |
