# GX教育项目交付管理系统 - Git提交规范

**版本**：V1.0
**日期**：2026年7月


## 一、提交信息格式

### 1.1 标准格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 1.2 示例

```
feat(auth): 添加JWT刷新令牌功能

- 实现刷新令牌接口 /api/v1/auth/refresh
- 添加令牌过期自动刷新机制
- 更新前端拦截器支持刷新逻辑

Closes #123
```


## 二、Type 类型说明

| Type         | 说明                   | 示例                                 |
| ------------ | ---------------------- | ------------------------------------ |
| **feat**     | 新功能                 | `feat(user): 添加用户导出功能`       |
| **fix**      | Bug修复                | `fix(login): 修复密码验证失败问题`   |
| **docs**     | 文档变更               | `docs: 更新API接口文档`              |
| **style**    | 代码格式（不影响功能） | `style: 统一缩进为2空格`             |
| **refactor** | 代码重构               | `refactor(db): 优化数据库连接池配置` |
| **perf**     | 性能优化               | `perf(api): 优化列表查询性能`        |
| **test**     | 测试相关               | `test(auth): 添加登录接口单元测试`   |
| **chore**    | 构建工具/依赖更新      | `chore: 升级FastAPI到0.109.0`        |
| **ci**       | CI/CD配置              | `ci: 添加GitHub Actions自动部署`     |
| **revert**   | 回退提交               | `revert: 回退feat(auth)提交`         |

## 四、Subject 规范

### 4.1 规则

1. **使用中文或英文**（团队统一），建议使用中文
2. **不超过50个字符**
3. **使用祈使句**（如"添加"而非"添加了"）
4. **首字母小写**（英文）或正常中文
5. **结尾不加句号**

### 4.2 好的示例

```
feat(auth): 添加JWT刷新令牌功能
fix(login): 修复密码验证失败问题
docs: 更新部署手册
refactor(db): 优化数据库查询性能
```

### 4.3 不好的示例

```
添加了新功能           # 缺少type
feat: 修复了一个Bug并添加了新功能  # 一次提交做了多件事
fix login              # 缺少详细描述
feat(auth): 添加了刷新令牌的功能。  # 结尾有句号
```


## 五、Body 规范

### 5.1 何时需要 Body

- 提交内容较复杂，需要详细说明
- 需要解释"为什么"做这个改动
- 需要说明与之前代码的关系

### 5.2 规则

1. 每行不超过72字符
2. 说明"是什么"和"为什么"，而非"怎么做的"
3. 使用项目符号列表（`-` 或 `*`）

### 5.3 示例

```
fix(db): 修复MySQL连接超时问题

- 增加连接池大小，解决并发请求时的连接等待
- 添加连接重试机制，提高容错性
- 将连接超时时间从5秒调整为30秒
- 修复SSL连接参数配置错误

Closes #45
```


## 六、Footer 规范

### 6.1 关联 Issue

```
Closes #123
Fixes #456
Refs #789
```

### 6.2 破坏性变更标记

```
BREAKING CHANGE: API响应格式从 {data} 变更为 {code, data, msg}
```


## 七、分支命名规范

| 分支类型 | 命名格式          | 示例                      |
| -------- | ----------------- | ------------------------- |
| 主分支   | `main` / `master` | 生产环境代码              |
| 开发分支 | `develop`         | 集成测试分支              |
| 功能分支 | `feature/功能名`  | `feature/user-management` |
| 修复分支 | `fix/问题描述`    | `fix/login-error`         |
| 热修复   | `hotfix/问题描述` | `hotfix/security-patch`   |
| 发布分支 | `release/版本号`  | `release/v2.1.0`          |


## 八、提交频率

### 8.1 原则

1. **原子性**：每次提交只做一件事
2. **完整性**：提交的代码应该是可运行的
3. **及时性**：完成一个小功能就提交，不要积累太多

### 8.2 提交粒度参考

| 场景             | 提交粒度  |
| ---------------- | --------- |
| 修复一个Bug      | 1次提交   |
| 添加一个API接口  | 1次提交   |
| 添加一个完整页面 | 1-2次提交 |
| 重构一个模块     | 1次提交   |
| 更新文档         | 1次提交   |


## 九、常用命令

### 9.1 基本操作

```bash
# 查看状态
git status

# 查看提交历史
git log --oneline

# 查看提交历史（详细）
git log --graph --pretty=format:'%Cred%h%Creset -%C(yellow)%d%Creset %s %Cgreen(%cr) %C(bold blue)<%an>%Creset' --abbrev-commit

# 添加修改
git add <file>          # 添加单个文件
git add .               # 添加所有修改

# 提交
git commit -m "type(scope): subject"

# 推送到远程
git push origin <branch>
```

> 📌 **push 执行约定**
> 本地与服务器均已配置 GitHub SSH 认证，`git push` 可由 **Claude 直接执行**。
> **约束：push 前必须先向用户说明将推送的提交内容并获得确认**（对外、影响远程仓库的操作，不可静默执行）。
> 说明历史：早期曾误认为"Claude 无 GitHub 认证、push 须用户手动执行"——该假设错误，已作废。

### 9.2 修改提交

```bash
# 修改最后一次提交信息
git commit --amend -m "新的提交信息"

# 合并多个提交
git rebase -i HEAD~3    # 合并最近3个提交
```

### 9.3 撤销操作

```bash
# 撤销工作区修改
git checkout -- <file>

# 撤销暂存区修改
git reset HEAD <file>

# 回退到某个提交
git reset --hard <commit-hash>
```
