## 🎯 复盘目标

1. **识别重复问题**:出现≥2次的问题,说明预防机制失效
2. **分析根本原因**:不只修表面,找到为什么会发生
3. **建立预防机制**:规范、工具、流程,避免同类错误
4. **趋势分析**:问题数量、严重度、修复时间的变化趋势

---

## ⏰ 复盘周期

- **频率**: 每2周一次(双周五下午)
- **时长**: 1-2小时
- **参与人**: GX-PM Team + Claude

## 📝 复盘流程

### 步骤1: 收集问题清单(复盘前3天开始)

**数据来源**:
- 开发日志(每日工作记录)
- Git commit message(搜 `fix`、`bug`、`hotfix`)
- 用户反馈(邮件、群消息、issue)
- 生产告警(错误日志、监控告警)

**收集工具**:
```bash
# 示例:搜索本周期的fix commit
git log --since="2 weeks ago" --grep="fix\|bug" --oneline
```

### 步骤2: 分类统计

按以下维度分类:
- **类型**: 验证不足、数据不一致、演示代码、编码、配置、其他
- **严重度**: P0(严重)/P1(重要)/P2(一般)/P3(轻微)
- **是否重复**: 本期≥2次,或与历史问题相同

### 步骤3: 识别重复问题(★重点)

**判断标准**:
- 本周期内出现≥2次
- 与历史复盘报告中的问题**根因相同**(即使表现不同)

**示例**:
- "只curl不做浏览器验证" 和 "没发现NaN" → 同一根因:验证不足
- "ddl.sql的type值" 和 "Device字段名" → 同一根因:ddl vs models不一致

### 步骤4: 深度分析根因

对每个重复问题,回答5个问题:
1. **What**: 具体发生了什么?
2. **When**: 什么时候发生的?有什么触发条件?
3. **Why**: 为什么会发生?(技术原因、流程漏洞、人为疏忽?)
4. **Impact**: 造成什么损失?(时间、体验、数据)
5. **How to prevent**: 如何预防?(规范、工具、流程)

### 步骤5: 制定优化建议

**分优先级**:
- P0(本周): 严重且紧急,立即修复
- P1(本月): 重要,1-2周内修复
- P2(本季度): 优化项,可延后

**要求**:
- 每条建议必须**可执行**(有具体脚本/流程/负责人)
- 每条建议必须**可验证**(如何检查是否执行了?)

### 步骤6: 制定行动项

| 编号 | 行动项 | 负责人 | 截止日期 | 验收标准 |
|------|--------|--------|---------|---------|
| A1 | 写Playwright验证脚本 | Claude | 2026-07-24 | 脚本能跑通关键路径 |
| A2 | 引入Alembic迁移工具 | GX-PM | 2026-07-31 | models改动自动生成迁移 |

### 步骤7: 撰写复盘报告

- 复制 `复盘模板.md`,改名为 `复盘报告-YYYY-MM-DD.md`
- 按模板填写各章节
- 保留所有章节结构(即使某章节为空)

### 步骤8: 追踪行动项执行

- 下次复盘时,首先检查上次行动项完成情况
- 未完成的行动项,分析原因并重新排期

## 📊 复盘报告结构说明

### 必填章节

1. **执行摘要**: 一页纸总结(问题数、核心收获)
2. **问题分类统计**: 按类型、严重度统计
3. **重复问题深度分析**: 每个重复问题单独一节
4. **全部问题清单**: 按时间线列表
5. **优化建议汇总**: 分P0/P1/P2优先级
6. **复盘行动项**: 具体任务+责任人+截止日期

### 选填章节

7. **经验教训**: 做对的、需改进的、新原则
8. **下次复盘计划**: 时间、范围、流程
9. **附录**: 数据统计、趋势对比

## 🔍 如何使用历史复盘报告

### 新问题出现时

1. **搜索历史报告**: `grep -r "关键词" docs/09-项目复盘/`
2. **检查是否重复**: 历史是否有相同/相似问题?
3. **查看预防机制**: 历史建议的预防措施执行了吗?为什么失效?

### 制定新规范时

1. **查阅历史教训**: 这类问题历史发生过几次?
2. **参考历史建议**: 之前提出的方案为什么没执行?还适用吗?
3. **避免重复建议**: 不要提已经建议过但未执行的措施(要问为什么未执行)

### 趋势分析

每季度做一次**跨期对比**:
- 问题总数趋势(上升/下降?)
- P0严重问题趋势
- 重复问题比例趋势
- 平均修复时间趋势

## ✅ 复盘质量检查清单

一份合格的复盘报告应该满足:

- [ ] 识别了所有重复问题(≥2次)
- [ ] 每个重复问题都做了5W分析(What/When/Why/Impact/How to prevent)
- [ ] 优化建议都是**可执行的**(有具体脚本/流程)
- [ ] 行动项都有**负责人和截止日期**
- [ ] 与历史报告做了对比(趋势分析)
- [ ] 保留了完整的章节结构(便于对比)

## 💡 复盘最佳实践

### ✅ DO

1. **追根溯源**: 问5次"为什么",找到根本原因
2. **量化损失**: 每个问题评估时间成本+业务影响
3. **预防为主**: 重点不是"怎么修",而是"怎么防"
4. **保留历史**: 所有报告保留,便于趋势分析
5. **行动导向**: 每条建议都要落实到具体行动项

### ❌ DON'T

1. **不要只列问题不分析**: 列100个问题没用,要找出重复的深挖
2. **不要只说"要注意"**: "以后要注意XX" 不是预防措施,要有具体工具/流程
3. **不要指责个人**: 复盘是改进流程,不是追责
4. **不要形式主义**: 不要为了复盘而复盘,要真正改进
5. **不要删除历史报告**: 历史是趋势分析的基础

---

## 📈 成功指标

复盘体系是否有效,看这3个指标:

1. **重复问题比例下降**: 每期重复问题占比应逐渐下降
2. **P0严重问题减少**: 严重问题数量应逐渐减少
3. **行动项完成率提升**: 上期行动项完成率应≥80%
