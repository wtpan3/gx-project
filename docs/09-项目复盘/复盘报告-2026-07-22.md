## 三、重复问题深度分析(★重点)

### 🔴 问题1: 验证不足——只做L2(curl)不做L3(浏览器)

**出现次数**: 3次
**具体案例**:
1. 2026-07-20: 部署后只curl测接口200,未发现CORS导致登录失败
2. 2026-07-22: 接口返回200,未发现前端显示NaN%
3. 2026-07-22: Claude说"菜单没做",实际只是没验证就断言

**根本原因**:
- 没有建立明确的验证标准(什么叫"完成"?)
- curl 简单快速,容易产生"测通了就完成"的错觉
- 没有自动化端到端测试,全靠人工抽查

**损失评估**:
- 每次误判"部署成功"浪费 30-60分钟排查
- 用户看到错误页面/NaN,体验极差
- 重复部署增加操作风险

**优化建议**:
1. ✅ **已建立**: DoD铁律——L3浏览器验证才算完成(已写入记忆+文档)
2. **待实施**: 写一个 Playwright 自动化验证脚本,部署后自动跑
3. **待实施**: 部署清单里加"L3验证通过"checkbox,未勾选不算完成

**预防机制**:
```bash
# 部署后强制执行的验证脚本
./scripts/verify_deployment.sh production
# 内部调用 Playwright 跑关键路径:登录→Dashboard→用户管理→退出
```

### 🔴 问题2: ddl.sql 和 models.py 不一致

**出现次数**: 3次
**具体案例**:
1. devices.type: ddl写'服务',models写'其他'
2. devices.status: ddl有'已发货'/'已验收',models有'运行中'
3. Device模型字段名: models写`installation_date`,数据库列名`install_date`

**根本原因**:
- ddl.sql 和 models.py 独立维护,无自动同步
- 需求变更时改了代码没改文档,或反之
- 没有"单一权威数据源"(SSOT)原则

**损失评估**:
- 插入演示数据失败(ENUM值不匹配)
- 潜在500错误(字段名不匹配,未来查询会报错)
- 3小时排查时间

**优化建议**:
1. **定权威**: models.py 是权威(代码是运行的真相),ddl.sql 是派生文档
2. **自动生成**: 用 Alembic 迁移工具自动从 models.py 生成 ALTER 脚本
3. **CI检查**: PR时自动对比 models.py 和最新迁移脚本,不一致则fail
4. **定期审计**: 每月跑一次"生产库 vs models.py"结构对比

**预防机制**:
```python
# 每次改 models.py 后自动生成迁移
alembic revision --autogenerate -m "sync models"
# CI 检查是否有未提交的迁移
if [ -n "$(alembic check)" ]; then echo "models.py changed but no migration!"; exit 1; fi
```

### 🔴 问题3: 演示代码/硬编码数据上线

**出现次数**: 2次
**具体案例**:
1. dashboard_service.py 硬编码 `装修中3/安装中2`,导致 `待启动=total-5=-2`
2. dashboard_service.py 假设 devices 表有数据,生产空表导致 `0/0=NaN`

**根本原因**:
- "先写演示数据快速验证UI"是合理开发策略
- 但演示代码**没有标记**(`# DEMO`),没人知道要改
- 没有上线前检查清单(`grep DEMO`)

**损失评估**:
- 生产显示 NaN%、负数,用户体验极差
- 2小时排查+修复时间

**优化建议**:
1. ✅ **已建立**: 演示代码规范(必须用 `# ⚠️ DEMO` 标记,写入文档+记忆)
2. **待实施**: pre-commit hook 检查: `git diff --cached | grep -i demo && echo "还有DEMO代码!"`
3. **待实施**: 部署清单: `grep -r "DEMO\|TODO: PRODUCTION" backend/ frontend/` 必须无输出
4. **前端防御**: 所有除法/数组访问加防御(即使后端是演示数据)

**预防机制**:
```bash
# .git/hooks/pre-commit
if git diff --cached | grep -iE '(⚠️ DEMO|TODO: PRODUCTION)'; then
  echo "❌ 发现演示代码标记,禁止提交到main分支"
  exit 1
fi
```

### 🟡 问题4: MySQL中文ENUM双重编码

**出现次数**: 4个表受影响
**具体案例**:
- device_systems.type
- software_modules.phase
- todos.priority / todos.status

**根本原因**:
- 导入SQL时未指定 `--default-character-set=utf8mb4`
- UTF-8字符被当作Latin1再转UTF-8,变成双重编码

**损失评估**:
- 4个表的ENUM无法正常显示/插入
- 1小时逐个表ALTER修复

**优化建议**:
1. ✅ **已写入文档**: 每次导入SQL必须加 `--default-character-set=utf8mb4`
2. **待实施**: 写一个 `scripts/import_sql.sh` 封装正确命令,禁止直接敲
3. **待实施**: ddl.sql 开头加注释: `-- 导入命令: mysql --default-character-set=utf8mb4 < ddl.sql`

**预防机制**:
```bash
# scripts/import_sql.sh (封装正确命令)
docker exec -i gx_mysql mysql -uroot -p"$MYSQL_ROOT_PASSWORD" \
  --default-character-set=utf8mb4 "$DB_NAME" < "$1"
```

### 🟡 问题5: "凭假设断言,不先验证"

**出现次数**: 2次
**具体案例**:
1. Claude说"Claude无git push认证,须用户手动push"(从未验证过,纯臆测,后来发现能push)
2. Claude说"导航菜单没做"(实际已做好,只是没读代码就下结论)

**根本原因**:
- 凭记忆/常识推断,而不是先查证
- 假设的结论被写进记忆,变成"事实"误导后续

**损失评估**:
- 错误信息误导3天
- 浪费时间在错误方向上

**优化建议**:
1. ✅ **已建立**: "验证后再断言"记忆规则
2. **执行要求**: 涉及"能/不能、有/没有"的断言,必须先 grep/Read/执行 验证
3. **记忆审计**: 每次写记忆前,再问一次"这是我验证过的事实,还是推测?"
