# 问题登记与复盘工作流

本文档定义**问题发生后的标准处理流程**,确保每个问题都被记录、分析、预防。

---

## 工作流总览

```
┌─────────────┐
│ 问题发生    │
└──────┬──────┘
       │
       ▼
┌─────────────────────────────────┐
│ 1. 立即修复问题                  │
│    - 排查根因                    │
│    - 实施修复                    │
│    - 验证(必须L3浏览器验证)      │
└──────┬──────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│ 2. 登记到问题登记簿(30秒-3分钟)  │
│    python scripts/log_issue.py  │
│    或手动编辑CSV                 │
└──────┬──────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│ 3. 每周五自动分析               │
│    自动生成复盘报告              │
│    识别重复问题                  │
└──────┬──────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│ 4. 人工审阅+制定行动项          │
│    补充根因分析                  │
│    制定预防措施                  │
│    分配责任人                    │
└──────┬──────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│ 5. 执行行动项                   │
│    下次复盘检查完成情况          │
└─────────────────────────────────┘
```

---

## 详细步骤

### 步骤1: 发现并修复问题(当下)

**操作**: 
1. 排查问题根因(不要只修表面)
2. 实施修复方案
3. **验证修复效果**(必须做L3浏览器端到端验证,不能只curl)
4. 提交代码: `git commit -m "fix(模块): 问题描述"`

**时长**: 根据问题复杂度,1分钟-数小时

**产出**: 
- 修复代码(已提交)
- 对根因的理解

---

### 步骤2: 登记问题(修复后立即,30秒-3分钟)

**为什么要登记**:
- 不登记 = 复盘时凭记忆回忆 = 容易遗漏
- 登记 = 积累数据 = 识别重复问题 = 真正预防

**操作方式A: 命令行快速登记(推荐)**

```bash
cd /path/to/gx-project
python scripts/log_issue.py
```

交互式问答:
```
问题描述(一句话): 生产Dashboard显示NaN%
严重度(0-3): 0
问题类型(1-6): 3
根因分析: devices表为空导致hardware_total=0
是否重复问题?: n

✅ 已登记问题 P019
```

**操作方式B: 手动编辑CSV**

打开 `docs/09-项目复盘/问题登记簿.csv`,追加一行:

```csv
2026-07-22,P019,Dashboard显示NaN%,P0,演示代码+空表,devices表为空,2.0,#123,已修复,否
```

**必填字段**:
- 日期、问题编号、问题描述、严重度、类型、根因

**选填字段**(修复后补):
- 修复时间、修复PR、状态

**时长**: 30秒-3分钟

**产出**: 
- 问题登记簿.csv 新增一行

---

### 步骤3: 每周自动分析(周五下午,自动)

**触发方式**:
- **自动**: Cron定时任务每周五15:00自动执行
- **手动**: `python scripts/analyze_issues.py --days 7`

**脚本做什么**:
1. 读取问题登记簿.csv
2. 筛选最近7天的问题
3. 统计分类(类型、严重度、重复问题)
4. 识别重复问题(同类型≥2次)
5. 生成初步复盘报告

**时长**: 自动,秒级

**产出**: 
- `docs/09-项目复盘/复盘报告-YYYY-MM-DD-自动生成.md`

---

### 步骤4: 人工审阅报告(周五下午,30-60分钟)

**操作**:
1. 打开自动生成的报告
2. **重点关注"重复问题"章节**
3. 补充每个重复问题的:
   - 详细根因分析(问5次"为什么")
   - 优化建议(具体可执行的措施)
   - 预防机制(工具/流程/规范)
4. 制定行动项(任务+责任人+截止日期)
5. 与上期复盘报告对比趋势

**产出**:
- 完善的复盘报告(去掉"自动生成"后缀)
- 行动项清单

---

### 步骤5: 执行行动项(持续)

**操作**:
1. 按优先级执行行动项(P0本周、P1本月、P2本季度)
2. 执行完成后更新状态
3. 下次复盘时检查完成率

**跟踪方式**:
- 在复盘报告的"行动项"表格里勾选完成
- 或在问题登记簿.csv里更新状态

---

## 关键原则

### ✅ DO

1. **问题修复后立即登记**,不要拖到复盘时
2. **根因分析要追问"为什么"**,不要只说"配置错了"
3. **重复问题必须制定预防措施**,否则会一直重复
4. **行动项必须可执行**,有具体脚本/流程/负责人
5. **每次复盘对比趋势**,看重复问题比例是否下降

### ❌ DON'T

1. **不要只修不记**,修完就忘 = 下次重复
2. **不要复盘时凭记忆回忆**,容易遗漏
3. **不要只列问题不分析**,列100个没用,要找重复的
4. **不要只说"以后注意"**,要有具体工具/流程
5. **不要形式主义**,为了复盘而复盘

## 脚本使用

### log_issue.py - 问题登记脚本

```bash
# 交互式登记
python scripts/log_issue.py

# 输出示例
问题描述(一句话): Dashboard显示NaN%
严重度(0-3): 0
问题类型(1-6): 3
根因分析: devices表为空
是否重复问题?(y/N): n

✅ 已登记问题 P019
```

### analyze_issues.py - 自动分析脚本

```bash
# 分析最近7天(默认)
python scripts/analyze_issues.py

# 分析最近14天
python scripts/analyze_issues.py --days 14

# 输出示例
=== 分析最近7天的问题 ===

问题总数: 5
类型分布: {'验证不足': 2, '数据不一致': 2, '演示代码问题': 1}
严重度分布: {'P0': 1, 'P1': 3, 'P2': 1}
重复问题: 3个(60.0%)
平均修复时间: 1.2h

✅ 复盘报告已生成: docs/09-项目复盘/复盘报告-2026-07-22-自动生成.md
```

## 成功指标

复盘体系是否有效,看这3个趋势:

1. **重复问题比例下降**
   - 第1期: 28%
   - 第2期: 目标<20%
   - 第3期: 目标<15%

2. **P0严重问题减少**
   - 第1期: 3个
   - 第2期: 目标<2个
   - 第3期: 目标0个

3. **行动项完成率提升**
   - 目标: ≥80%

## 常见问题

**Q: 小问题也要登记吗?**
A: 是的。即使是P3轻微问题也要登记,因为同类型出现≥2次就是重复问题,说明预防机制失效。

**Q: 登记花多久?**
A: 用脚本30秒,手动编辑CSV 1-2分钟。比起下次重复踩坑的成本,这时间很值得。

**Q: 复盘报告谁来写?**
A: 脚本自动生成初稿,PM或Claude审阅补充根因分析和行动项。

**Q: 如果忘记登记怎么办?**
A: 补登记,在CSV里补上历史问题。但尽量养成修复后立即登记的习惯。

**Q: 行动项没完成怎么办?**
A: 下次复盘时分析原因:是任务太大?责任人忙?还是优先级评估错误?调整计划。
